# Brahmand — E2E Chain Execution Engine

**Status:** 🟢 Live paper trading | **Updated:** 2026-05-21

Brahmand is the **execution layer** of the automated trading system. It runs a 5-agent sequential pipeline that transforms entry signals into live trades with continuous risk monitoring.

## How It Works

### Entry Point: `kickoff.py`

Runs **every 5 minutes** during market hours (9:15-15:30 IST):

1. **Lock protection** — prevents overlapping runs
2. **First run of day** — enters a new trade via E2E Chain
3. **All runs** — monitors active trades, applies TSL, handles exits
4. **State persistence** — saves to `data/brahmand_kickoff.json` and `state.db`

### Pipeline: E2E Chain (`e2e_chain.py`)

Sequential CrewAI pipeline with 5 agents:

```
Entry Agent     → Read EMA trend + traffic light signal
    ↓
Regime Agent    → Validate signal against live market data
    ↓
Strategy Agent  → Select strategy, calculate strikes, SL/TP
    ↓
Contract Resolver (Python) → Resolve tsyms from option snapshots
    ↓
Executor (Python) → Place 4-leg trade, save to state.db
    ↓
Risk Monitor    → 1-min cadence, TSL, exits
```

## Key Concepts

### Trailing Stop Loss (TSL)

- Activates when profit ≥ 25% of max TP
- Ratchets SL downward on favorable moves
- Locks in gains progressively
- Only improves SL, never worsens it

### Position Morphing

When a position should shift to new strikes:
- Wing too close (delta > 0.30)
- Time decay insufficient
- VIX spike/drop (> 25% change)

See `position_manager.py` for detection logic.

### State Persistence

Two storage layers:

1. **JSON** (`data/brahmand_kickoff.json`) — Daily state, fast reads
2. **SQLite** (`state.db`) — Trade history, orders, monitoring for analytics

## FAQ

**Q: Why does kickoff run every 5 minutes?**  
A: Allows rapid re-entry after trade closes, responsive to market changes, minimal execution delay.

**Q: What if a run overlaps with the previous one?**  
A: Lock protection (PID-based) prevents overlapping. Previous run must complete within 5 minutes.

**Q: How is P&L calculated?**  
A: Per-leg: `(entry_price - ltp) * qty` for BUY, `(ltp - entry_price) * qty` for SELL.  
Total: sum across all 4 legs.

**Q: What triggers a position exit?**  
A: SL breach, hard time limit (15:20), manual override, data staleness.

**Q: Can I manually intervene?**  
A: Yes, use the emergency exit function or Telegram override.

**Q: Where are logs stored?**  
A: `logs/kickoff_YYYYMMDD.log` (persistent, not in /tmp).

**Q: How do I check if EMA is properly wired?**  
A: After the fix, run:
```bash
python3 -c "from ema_aggregator import load_ema; print(load_ema('NIFTY', 1))"
# Should print fresh EMA array
```

## Troubleshooting

**Issue: "DuckDB empty"**  
Solution: Check `data_capture_v3_duckdb.py` is running. Verify `varaha_data.duckdb` file exists.

**Issue: "EMA files not found"**  
Solution: Run `ema_backfill.py` to recover from v3.1 data. Verify `ema_state/` directory accessible.

**Issue: "No entry signal"**  
Solution: Check `entry_check.py` is executing. Verify `entry_signals/entry_check_latest.json` exists and is fresh.

**Issue: "Lock timeout"**  
Solution: Previous kickoff run exceeded 5 minutes. Check logs for slow operations. Increase timeout or optimize.

**Issue: "Contract resolution fails"**  
Solution: Check option_snapshots table has latest data. Verify strike exists with sufficient liquidity.
